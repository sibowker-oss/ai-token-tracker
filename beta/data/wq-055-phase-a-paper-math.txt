## Critical methodology question — flagged for Simon

Two interpretations of resolve_opex TIER 2 condition. The pseudocode in §3.1:

    if op_loss is not None and cr is not None and inf is not None:
        ...

reads `inf = fin.get("inference_cost")` (entity field). But the
math comment "opex = (op_loss + ref_revenue) - inf" implies
the resolved inference (which can be sourced/derived/estimated).

  Interpretation A (LITERAL pseudocode): TIER 2 of opex requires
    entity-field `inf` to be present. Otherwise falls to TIER 3.
    For OpenAI, both interpretations give same result ($5B is sourced
    AND in entity field). For Anthropic this matters: inf would have
    to be entity-field-present (it isn't), so opex TIER 2 fails →
    falls to TIER 3 size_band estimate.

  Interpretation B (LOOSE reading): TIER 2 uses resolved inf
    regardless of source tier. For Anthropic this would fire TIER 2
    with inf=2.59 (estimated) and ref_revenue=4.5 → opex=$7.11B →
    provider_total=$9.70B (+25.8% vs hand-curated $7.71B, OUTSIDE band).

Math below uses interpretation A. Flagging this for Simon's confirm
because Anthropic's outcome flips dramatically between the two.

    R&D heaviness, would push provider_total higher and bring it nearer

  - $5-15B at 0.90× captures "scaling efficiency" — directionally right
  - $15-50B / $50B+ untested in 2025 data (all model_providers below $15B cr)

So no single band tuning fixes both Anthropic and Google. Could need
provider-specific opex rate overrides for Google (the brief's override
