## Decisions made during implementation

1. **Frontier-lab `enterprise_pct → Hyperscalers` weight = 50% (`_default` row).** Brief D2 specified weights for AI Native, Enterprise SaaS, and the API side of frontier_lab but left frontier_lab `enterprise_pct` unspecified. Choosing 50% (i.e. fall through to `_default`) lands Hyperscalers at $1.70B — squarely inside D4's $1.5–2B band. Adding an explicit frontier_lab row at lower weight (e.g. 30%) would push Hyperscalers below $1.2B, outside the brief's implied-pass-through target $1.2–1.6B.
2. **`_default_for_unmapped` channel split kept unchanged.** The wq-062 default for entities with no `revenue_by_channel` (`{Subs:0.4, API:0.4, AINative:0.2}`) still applies. Per the brief's edge-case rule ("under-counting Hyperscalers is editorially safer"), this default has zero Hyperscalers weight, which is correct.
3. **Buyer-segment math runs in gross terms.** `derive_buyer_segments` applies channel margins (1/(1−margin)) per split so the buyer-gross totals tie to `sum(channels.gross)` by construction — preserving the buyer↔channel conservation identity the renderer relies on.
4. **`generate_site_data.py` rescales buyer segments to match `sum(channels.gross)`** before writing. Engine math has small float drift between `buyer_segments_gross` and `channels_grossed`; the rescale forces exact conservation without distorting the within-buyer ratios.
5. **Backward-compat colour aliases on revenue.html.** Older data files referencing SME/Enterprise (e.g. older snapshots in `data/sankey-projections.json` projection narratives) still resolve to the new palette — partial deploys don't break colours.
6. **Cohort spec preserved.** Sankey-relevant entities are still `model_provider` + the IaaS rollup. `entity_archetype` is added to *every* entity in entities.json so downstream `derive_market_aggregates.py` can apply per-archetype segment composition consistently across the cohort.

At Tier 2A precision the two sit comfortably inside reconciliation tolerance.
