## What shipped

A single canonical apply script (`scripts/apply_pipeline.py`) replacing the
legacy `apply_claims.py` + `apply_decisions.py` pair. The new pipeline reads
`vault-data.json` directly, dispatches every claim to a registered typed
handler keyed on the claim's `unit`, writes resolved values to
`entities.json` (which `generate_site_data.py` consumes for
`entityDirectory` + `arrModel`), and is idempotent.

### New
- `scripts/apply_pipeline.py` — canonical apply script. Implements:
  - Handler-registry dispatch by `unit`
  - Auto-apply gate (D2): `confidence='verified'` AND tier in
    `{tier_1A, tier_1B, tier_2A}`. Tier 2B+ stays in vault for wq-100
    review UI.
  - D6 conflict resolution: highest tier wins; ties → latest dateOfClaim
    → latest dateAdded. Also implements within-run dedup so multiple
    claims targeting the same `(entity, period, field)` collapse to a
    single winner before commit.
  - D8 idempotency: any non-empty `usedOn` skips the claim. Re-running
    `python3 scripts/apply_pipeline.py --dry-run` produces zero pending
    changes (verified).
  - D9 material-change gate at $5B aggregate ARR delta on
    ai_app + model_provider entities.
  - Acceptance #9 divergence guard: existing values within ±15% of
    incoming get overwritten; >15% divergence keeps existing value but
    extends the provenance trail (logs WARN).
  - Compute-disclosure evidence reconciliation for hyperscaler ARR
    claims whose value already lives in
    `data/evidence/compute_disclosures/*.json` (covers acceptance #6
    Microsoft AI $37B path).
  - Audit doc generation at `data/audits/wq-098-skipped-claims.md` so
    every skipped claim is auditable.
- `scripts/apply_handlers/` — typed-handler package with one module per
  claim category:
  - `arr.py` — ARR / annualised revenue ($B / USD billions / $B ARR /
    USD millions / $M ARR / $B ARR (peak-4wk))
  - `valuation.py` — equity / pre-money / IPO valuation
  - `acv.py` — annual contract value ($M ACV / $B ACV)
  - `pricing.py` — list-price subscription + per-token claims
  - `user_count.py` — WAU / MAU / seats / business customers
  - `employee_count.py` — headcount + AI-titled hiring
  - `infrastructure.py` — GW / kW / GPUs / $/GW
  - `quarterly_revenue.py` — quarterly revenue → annualised; T tokens
  - `burn.py` — cash burn / operating loss / cumulative burn
  - `growth.py` — % YoY / x growth / drawdown
  - `telemetry.py` — downloads / version numbers / SEC filings catalogue
    (recorded but not propagated; wq-099 reroutes)
  - `_default.py` — fallback for units without a dedicated handler;
    fails CI if a vault claim's unit is neither registered nor in
    `UNHANDLED_UNITS`
  - `_shared.py` — entity resolution, tier derivation, value coercion,
    conflict-resolution helpers
- `scripts/validate-pipeline-orphans.mjs` — CI assertion that
  - every vault `unit` has a registered handler
  - no verified tier-2A-or-better claim sits unused
  - only allowlisted scripts write `vault-data.json`
  - no production code invokes `apply_*_legacy.main()`
- `data/audits/wq-098-direct-vault-writers.md` — Phase A.2 audit of
  every script that opens `vault-data.json` for write.
- `data/audits/wq-098-skipped-claims.md` — generated by every
  `apply_pipeline.py` run; lists each skipped claim with reason.
- `tests/test_apply_pipeline.py` — 22 unit tests covering tier
  derivation, auto-apply gate, value coercion, entity resolution,
  Cursor ARR handler, conflict resolution, idempotency, do_not_apply,
  and the units-coverage assertion.

## Decisions made during implementation

- **Underscore form for archived legacy scripts.** The brief specified
  `apply_claims.legacy.py` (dotted form). Python's import system can't
  load a dotted filename without a parent package, so the helper-only
  consumers (`audit_period_attribution.py`,
  `curated_intake.py`, `score_materiality.py`) couldn't import
  `detect_period_scope` etc. Switched to underscore form
  (`apply_claims_legacy.py`); same archival semantics, same CI fence
  (`validate-pipeline-orphans.mjs` checks no production code calls
  `apply_*_legacy.main()`).
- **Gap-fill / divergence-guard / overwrite ladder for D6.** Brief D6
  describes how to pick among competing claims, but doesn't spell out
  what to do with existing scalar values. I implemented the safer
  semantics that match acceptance #9 verbatim: append to provenance
  always; gap-fill when existing is None; overwrite when delta ≤ ±15%;
  WARN-and-skip-overwrite when delta > 15%. Matches the brief's
  "no silent overwrites" intent.
- **Stricter entity resolution (text-fallback narrowed).** The
  pre-rebuild `match_entity` regex would match "OpenAI Press Release"
  in `claim.notes` and route every Codex / SDK / valuation claim to
  the OpenAI entity. Restricted text fallback to
  `claim + tags + metricKey` (NOT notes / sourceAuthor) and added a
  product-disambiguation list (`Codex`, `Claude Code`, `GitHub Copilot`)
  that aborts resolve so the claim routes to audit. Caught dp-110
  (Codex 2M users) regression.
- **Compute-disclosure evidence reconciliation.** Acceptance #6 calls
  for dp-239 (Microsoft AI $37B) to "propagate to wherever the Compute
  Ledger MSFT figure lives." That value is already in
  `data/evidence/compute_disclosures/azure.json:arrGrossDisclosed`
  (hand-curated). Added a post-handler reconciliation pass that
  matches each vault ARR-shape claim against compute-disclosure
  values within ±10% and marks `usedOn=["evidence:compute_disclosures/<id>"]`
  without rewriting the evidence file. This matched dp-239 to
  `azure.json` exactly ($37B == $37B) on the first run.
- **Fallthrough for $B-but-not-ARR claims.** Many vault claims carry
  `unit: "$B"` but describe valuation, funding, or burn. The unit-keyed
  dispatch sends them to the ARR handler, which filters them out. Added
  a fallthrough in `process_claim` that, when the ARR handler returns
  `skip_reason='unit_matches_but_not_arr_claim'`, retries against the
  valuation or burn handler based on text signals
  (`valuation` / `valued at` / `cash burn` / `operating loss`). Caught
  ~12 OpenAI / Cursor / Perplexity valuation claims.
- **Telemetry handler returns `applied=True` with no entity write.**
  Telemetry signals (PyPI/npm/HF downloads, SDK versions, SEC filings
  catalogue) have no canonical home in `entities.json`. Per scoping doc
  Failure #2 these were direct-to-vault writes that should have been
  inbox-only. Pragmatic fix here: mark them `usedOn=["telemetry-signals"]`
  so the orphan validator stops pinging on them; wq-099 reroutes
  proper homes.
