## §0 — Context

**The defect.** TAIL's claims pipeline is structurally broken. 79% of vault data points (191 of 242) have `usedOn: []`. 16 of those are verified ARR claims >$100M that should be in the rendered site but aren't. Of 70+ distinct claim types in the vault, the apply pipeline handles only six. A `data-updates/approved-claims.json` file required for the apply step doesn't exist on disk and hasn't been created (claims.html outputs it as a browser download requiring manual file move). Multiple ingestion scripts write directly to `vault-data.json` bypassing the apply chain entirely.

The full diagnosis is in the scoping doc; this brief delivers the rebuilt apply layer. Companion briefs deliver the visibility surface (wq-099) and review-UI redesign (wq-100).

**What this brief delivers:** a single canonical apply script that handles every claim type, writes to the canonical consumer structures (`entityDirectory` + `arrModel`, not the legacy `dashboard.*` paths), runs against the full vault on a daily schedule, is idempotent, and refuses to silently propagate claims it doesn't understand.

## §1 — Goal

A single canonical `apply_pipeline.py` script handles every claim type in the vault, writes to `entityDirectory` (which `generate_site_data.py` consumes for `arrModel`), runs daily, is idempotent, and fails CI if a claim type appears without a registered handler. After backfill, zero verified tier-1A/1B/2A claims sit unused. After this brief: structural integrity restored across the apply chain, ready for wq-097 publication.

## §4 — Implementation outline

### Phase A — Audit & schema (1 session)

1. Enumerate every distinct `unit` and `tags` combination in vault-data.json. Map each to a handler category. Surface any combinations with no clear category as open decisions.
2. Audit every script that writes to `vault-data.json`. Document each in `docs/audits/wq-098-direct-vault-writers.md`. For each, decide: (a) redirect to vault-inbox.json + auto-classify, OR (b) keep direct write with explicit allowlist exception (mojibake fixers only).
3. Add `do_not_apply` field to vault schema; document in CLAUDE.md.

### Phase B — Build apply_pipeline.py + handlers (1.5 sessions)

1. Implement `apply_pipeline.py` core loop: load vault → for each unprocessed claim → dispatch by type → handler returns updated entityDirectory delta → apply → mark `usedOn` → commit batch.
2. Implement handler registry with explicit type→handler mapping.
3. Implement each handler in `scripts/apply_handlers/` per §2. Each handler:
   - Reads claim
   - Resolves entity (fuzzy-match per D12)
   - Determines target field on entityDirectory
   - Applies conflict resolution per D6
   - Returns delta object
4. Implement `--dry-run`, `--report`, `--confirm-major-change` flags.

### Phase C — Bypass closure (0.5 session)

1. For each script identified in Phase A audit, redirect direct vault-data.json writes to vault-inbox.json with `auto-applied: true` flag.
2. Run apply_pipeline.py to consume the redirected items immediately.
3. Add CI assertion: only `apply_pipeline.py` and explicitly-allowlisted maintenance scripts write to vault-data.json.

### Phase D — Backfill (0.5 session)

1. `python3 scripts/apply_pipeline.py --dry-run --verbose` — review the change set with Simon before committing.
2. **Decision point:** if dry-run shows >$5B aggregate change in `arrModel.combined.industry_total`, halt and surface in chat for explicit approval before proceeding.
3. Run for real. Commit `vault-data.json` + `site-data.json` + audit doc together.
4. Regenerate visual baselines for any pages where figures shifted.

### Phase E — Wire auto_update + validators (0.5 session)

1. Replace `apply_claims.py` invocation with `apply_pipeline.py` in `auto_update.py`.
2. Extend `release-check.mjs` per §3 acceptance #10 + #2.
3. Verify next-morning log shows clean run.

### Phase F — Verify & ship (0.5 session)

1. `npm run build-lint` passes.
2. `npm run release-check` passes including new assertions.
3. Manual eyeball of `arrModel.apps.aiNative` top-12 entries vs. current public reporting — figures match.

## §5 — Edge cases

1. **Claim entity references unknown company.** Surface in audit doc with claim ID + entity string. Do NOT auto-create entityDirectory entries — that's a separate workflow (entity surfacing). If audit accumulates, escalate to a Cowork decision.

2. **Multiple conflicting claims for same entity+field+period.** D6 conflict resolution applies. Document chosen claim ID in `arrSource`. If two claims have identical provenance + dateOfClaim + dateAdded, log warning and pick alphabetically by claim ID for determinism.

3. **Multi-tier company claims.** Per D12, claim's `entity` field must specify tier-record (e.g., `google_workspace_ai`). If the claim only says "Google", route to audit with "ambiguous tier" reason and let human disambiguate. Don't silently route to one tier.

4. **Claim has higher provenance than current entityDirectory value but different number.** Take the new claim. Log the change in apply log + audit doc.

5. **Claim has lower provenance than current entityDirectory value.** Skip. Don't overwrite a Tier 1B disclosed figure with a Tier 3A leak.

6. **Vault claim is decimal-incompatible.** E.g., `unit: "USD"` but `value: "approximately 2 billion"` (string not number). Handler should fail loudly, surface in audit, not silently coerce. coerce_date.py exists; equivalent for value coercion may need to be built.

7. **Visual regression risk.** This brief WILL change rendered numbers. Re-baseline visual snapshots in same deploy. Document in deploy record which pages shifted and by how much.

8. **Handler implementation gaps.** If during Phase B a handler turns out to need more design (e.g., GW infrastructure has nuances we didn't anticipate), file `docs/decisions/open/` rather than guessing. The brief priorities ARR, valuation, ACV, user count, quarterly revenue handlers; lower-priority handlers (telemetry, growth, burn) can ship in a follow-on if scope balloons.

## Addendum 2026-05-08 — D2 gate revision (hotfix)

After the brief shipped, Cowork investigation surfaced that
`is_auto_apply`'s `confidence='verified'` requirement was silently
blocking all `estimated`-confidence claims even after Simon accepted
them in `claims.html`. The blocked cohort included Sacra deep-dives,
The Information leaks, and other credible private-company reporting
that TAIL relies on for AI-native ARR coverage. Concrete victim:
`dp-195` (Perplexity $500M ARR, April 2026) sat with `usedOn: []` so
`arrModel.apps.aiNative.entries[Perplexity].arr` stuck at the legacy
$200M figure.

### Revision to D2

- A claim auto-applies if EITHER (a) it has been human-reviewed (
  `human_reviewed: True`, set during inbox migration when the source
  inbox item carried `status='accepted'`) AND its source-type-only
  tier reaches tier_2A; OR (b) it carries `confidence='verified'` AND
  the standard ladder reaches tier_2A.
- Human review IS the trust signal for accepted-then-estimated claims.
  The confidence field captures source-strength, not review-status.
- `derive_tier_for_gate` is the per-claim helper that promotes a
  human-reviewed claim's tier as if `confidence='verified'` —
  `web_page` + `estimated` + `human_reviewed` → `tier_2A` for the
  gate, instead of the natural `tier_3A`.

### New supporting changes (hotfix scope)

- `apply_pipeline.py:migrate_accepted_inbox_items` — restores the
  inbox→vault migration step the legacy `apply_decisions.py` used to
  perform. Stamps `human_reviewed: True` on minted and backfilled
  vault entries.
- `_variance_gate.evaluate` — bypasses for `human_reviewed` writes
  because the wq-100 anomaly band would otherwise lock-out
  legitimate units corrections.
- `wq096_emit.py` — vault-first ARR sourcing in
  `_ai_native_app_entries` and `_frontier_arr_entries`. Closes the
  hardcoded-`topConsumers` leak that was the originally intended D4
  cleanup but was deferred per
  `docs/decisions/open/dec-2026-05-09-d7-dashboard-block-removal.md`.
  The legacy fixture path remains as a fallback for entities without
  a record; reconcile assertion #8 surfaces those for follow-up.
- `reconcile_pipeline.py` — assertions #7 (inbox migration
  freshness) and #8 (arrModel vault-backed) catch the failure modes
  the hotfix is correcting.

### Decisions made tactically

- D2.1 (hotfix): "tier promotion" only for the gate, not for the
  vault claim's natural tier in audit/reporting contexts. Recorded
  as the `tier` field in `prov_entry` to keep D6 conflict resolution
  consistent across pre- and post-hotfix entries.
- D9.1 (hotfix): the material-change gate's `arr_delta` calculation
  collapses to `|incoming|` when prior is ≥100x larger than incoming
  AND incoming < $5B. Recognises a units-correction artifact (legacy
  $M-stored values being overwritten by correctly-normalised $B
  values from the apply pipeline) without suppressing legitimate
  large values.

Acceptance: `arrModel.apps.aiNative.entries[Perplexity].arr` = 0.5
($500M) sourced from `dp-195`. `arrModel.combined.industry_total`
moves +$0.30B. Reconcile assertion #7 currently surfaces 19 malformed-
unit inbox items deliberately skipped from the mint path (open work
to clean up at source); assertion #8 surfaces 16 ai_native_app
entries that need entity surfacing (open work).
